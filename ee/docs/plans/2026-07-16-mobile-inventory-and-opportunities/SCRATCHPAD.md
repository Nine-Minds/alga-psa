## Decisions

- (2026-07-16) **Barcode column lives on `service_catalog`**, next to `sku` (catalog data, not inventory-tracking data), partial-unique per tenant among products — same constraint shape as `service_catalog_product_sku_unique`. Rejected: `product_inventory_settings` (a UPC exists whether or not stock is tracked); per-unit barcode column (serial/MAC barcodes already encode the stored value).
- (2026-07-16) **Lookup resolution** (revised in critique round — was a strict priority order): run all four exact matches together (normalized barcode, SKU, serial, MAC case/separator-insensitive); >1 domain hit returns a `multi` variant (never silently shadow a unit); zero hits → none+candidates. One endpoint, discriminated union — mobile renders from `type`.
- (2026-07-16) **REST wraps shared package code, never re-implements**: house precedent is the opportunities module (`lib/opportunityDetail.ts` feeds both the server action and REST). Inventory REST service calls `packages/inventory` lib/model cores; interaction REST calls the packages/clients model + create helper. If an action's core is welded to `withAuth`, extract a transaction-aware core into the package lib (pattern: `completedActionInteraction.ts`, `opportunityWin.ts`).
- (2026-07-16) **Mobile scope for inventory v1**: lookup/stock/units read; manual receive; PO-line receive; adjust; blind counts (record+submit, approve stays web); transfer receive. Explicitly out: SO fulfillment, RMA, loaners, kits, vendors, PO creation — web surfaces exist and are manager work, not standing-at-the-shelf work.
- (2026-07-16) **Mobile scope for opportunities v1**: queue + pipeline + detail + complete-action + log interaction + call-log prompt + schedule follow-up + win/lose. Out: create opportunity, suggestions accept/dismiss, board, EE surfaces (forecast/meeting/commitments), AI drafting.
- (2026-07-16) **Call logging is prompt-on-resume, never automatic** (no CallKit/call-detection): record a pending-call context when `tel:` launches from a deal screen, prompt on `AppState` resume within a 4h window, duration prefilled from elapsed time, fully dismissible. `useAppResume` hook already exists.
- (2026-07-16) **Calendar = AlgaPSA schedule entries** via existing `/api/v1/schedules` (mobile already consumes it). Follow-up = `ad_hoc` entry assigned to self — same shape the discipline engine's solo interrupt uses. No schedule↔opportunity FK exists; v1 does NOT add one (title carries the deal name; deep linkage deferred — candidate: opportunity_evidence ref_type='schedule_entry' on create).
- (2026-07-16) **Section gating** (revised in critique round — the pre-auth `/api/v1/mobile/auth/capabilities` endpoint cannot carry per-user flags): new authenticated `GET /api/v1/mobile/me/capabilities` — product code (`psa`) + RBAC per section — plus a mobile session capabilities store and conditional drawer registration. Old-server/new-app degrades to hidden tabs (404 → flags false).
- (2026-07-16) **Workflow gaps scoped** (revised in critique round) to: business-ops module with 2 read + 2 write actions; SO/PO lifecycle catalog entries — with the missing `*_DELETED` emit sites added first (they were declared but never published); 4 new events (transfers ×2, counts ×2 — RMA_STATUS_CHANGED cut, unit events already cover most RMA transitions); STOCK_LOW emission parity on adjust/count-approve. Rejected for v1: loaner-overdue + dead-unit-owed aging events (need a scheduled job, not an emission point — note for a follow-up plan), stock-unit CRUD catalog entries (search-index churn, not business events).
- (2026-07-16) **Plan location**: `ee/docs/plans/` (house convention per sales-opportunity + inventory-psa-integration plans; user feedback says never OSS `docs/plans`).
- (2026-07-16) **Implementation logistics**: mechanical backend lanes (REST controllers/services/schemas, event seeds, mobile API modules + their tests) offload to codex via model-offload skill; mobile UI/UX, scan interactions, and all review stays with Claude. Commit per commitGroup with /commit-message; no Co-Authored-By tagline (user rule).

## Critique round 1 (2026-07-16) — findings folded in (revision 2)

Hostile review verified most research but found 1 blocker + 7 majors. All folded:

- **Core extraction was unbudgeted (BLOCKER)**: inventory actions hold logic inline in `withAuth` closures (receive `stockLedgerActions.ts:183-297`, adjust `:345-494`, all cycleCount, `receivePoLine` `purchaseOrderActions.ts:597-767`, stockUnitActions inline queries) — unlike opportunities' dual-consumer cores. Added FR2a + commit group `inventory-core-extraction` (F005–F009) with regression gate; it precedes all REST work. Verified session-free already: `assertLocationWritable(scope.ts:11)`, `createInteractionWithSideEffects`, `assembleWorkQueue`, `hasPermission` (re-fetches roles; works with API-key user).
- **Capabilities redesigned**: `/api/v1/mobile/auth/capabilities` is unauthenticated pre-auth config (providers/TTLs) held in SignInScreen local state — can't carry per-user flags. New authenticated `GET /api/v1/mobile/me/capabilities` + new mobile plumbing (session capabilities store + conditional drawer registration — drawer is fully static today). F050/F051.
- **GTIN normalization mandatory**: iOS maps upc_a→ean13 and strips the leading 0 (`BarcodeScannerUtils.swift:37-43,73-80`) — same label reads 12 digits on iOS, 13 on Android. One shared normalization function for storage/uniqueness/lookup; T003/T004. codabar needs iOS 15.4+.
- **Product gating corrected**: REST seam is `ApiBaseController.assertProductApiAccess`/`resolveProductApiBehavior` (NOT `enforceServerProductRoute`, which is RSC-only); unlisted API paths default-deny for AlgaDesk. Decision: `/api/v1/interactions` → `api_helpdesk_allowed` (core CRM AlgaDesk uses); inventory/opportunities stay PSA-only; explicit API_RULES entries for metadata correctness.
- **`receivePoLine` never scoped locations** (only checks existence at `:614`) — extraction adds `assertLocationWritable` (deliberate web behavior change), T007.
- **Test harness split**: house opportunity REST tests are 4 files/9 tests of schemas/registry/mocked-service — no auth/RBAC/DB. tests.json now tags every test `server-unit` / `server-integration` / `package-unit` / `mobile-vitest`; persistence/RBAC/constraint claims moved to the integration harness.
- **F103 enum trap**: schedules create schema has no `ad_hoc` in `work_item_type` enum (`timeSheet.ts:283`); service maps non-ticket/project to ad_hoc (`TimeSheetService.ts:1037-1043`); mobile type also excludes it. Fix: omit the field; correct mobile type.
- **Events honesty**: `INVENTORY_SALES_ORDER_DELETED`/`INVENTORY_PURCHASE_ORDER_DELETED` declared but never published — add emit sites before cataloging. RMA_STATUS_CHANGED cut (RMA out of scope; unit events cover most transitions; pure-status events → loaner/RMA-aging follow-up plan). Three touch points per event: schema file + `workflowEventPayloadSchemas` map + catalog seed (UNKNOWN_TRIGGER_EVENT class).
- **OpenAPI claim corrected**: opportunity routes ARE registered (`registerOpportunitiesV1Routes`, 27 endpoints); checked-in spec artifacts are stale. F042 = register new families + regen.
- **Interactions REST trimmed**: PUT/DELETE out of v1 (update/delete side effects inline in web actions; mobile doesn't need them). `interaction-types` must union `system_interaction_types` + tenant table (model method is tenant-only). Timeline needs a small core extraction first (query inline in withAuth).
- **RBAC list fixed**: + `cycle_count` (has extra `approve` action) and `stock_transfer`. Landmine: readd migration `20260707120000` omits `cycle_count` — verify seeded tenants, readd if missing (F036).
- **PO REST nested** under `/api/v1/inventory/purchase-orders` (receiving subset, not a public PO CRUD family).
- **Won on mobile**: calls the real win endpoint (close gates + conversion prep — NOT a plain status write); gate failures verbatim; success note "finish conversion on web".
- Camera copy: app.json plugin is source of truth; checked-in Info.plist already drifts — regenerate, don't hand-edit.
- Attribution caveat documented on `inventory.adjust_stock`: `performed_by` = workflow publisher (run-actor pattern, `shared.ts:104-129`).

## Discoveries / Constraints

- Expo SDK 54 / RN 0.81.5 / React 19, TS strict, **React Navigation drawer** (no expo-router), **plain hooks** (no react-query), **Vitest** (not Jest), react-test-renderer for component tests.
- Section = `Drawer.Screen` in `src/navigation/DrawerNavigator.tsx` + `DrawerParamList` in `src/navigation/types.ts`; detail screens in `RootNavigator.tsx`/`RootStackParamList`.
- API: `src/api/client.ts` (`createApiClient`) — Bearer + `x-api-key: session.accessToken` + `x-tenant-id` + `x-alga-client` tag; GET retry/dedupe; 401 refresh. One typed module per domain in `src/api/*` with colocated tests asserting method/path/headers. Mobile exchanges session → API key at `POST /api/v1/mobile/auth/exchange`.
- **Camera already provisioned**: `expo-camera` dep + plugins + iOS/Android permissions all present; working scan pattern in `src/screens/ServerEntryScreen.tsx` (`CameraView` + `useCameraPermissions` + `barcodeScannerSettings`); currently QR-only → add ean13/ean8/upc_a/upc_e/code128/code39/itf14/codabar.
- `tel:` via `Linking.openURL` already used in ClientDetail/ContactDetail. No expo-calendar/contacts/call-detection (and none needed).
- Templates to copy: list+detail = `ClientsListScreen`/`ClientDetailScreen`; CRUD form = `ScheduleScreen` + `ScheduleEntryFormModal`; picker = `EntityPickerModal`. States: `src/ui/states/*`. i18n: register namespace in `src/i18n/config.ts` + `i18n.ts` + `locales/en/*.json`.
- `src/api/materials.ts` already consumes `/api/v1/products` (has `sku`) — reuse types where sane.

### Inventory module — map digest
- `packages/inventory` (CE package, product-gated `psa` via `enforceServerProductRoute`, NOT license-gated). Tables `2026-06-26+`: stock_locations, product_inventory_settings, stock_levels, stock_units (serial NOT NULL unique per (tenant,service), MAC tenant-unique partial, both indexed), stock_movements (append-only), vendors, vendor_products, POs/SOs + lines, transfers, rma_cases, cycle counts (count_sessions/count_lines), vendor_bills, landed costs. Interfaces: `packages/types/src/interfaces/inventory.interfaces.ts`.
- Actions (all `withAuth` + `hasPermission`): stockLedgerActions (receiveStockManual/adjustStock/retireStock), stockUnitActions (searchUnitsBySerial/ByMac/searchInStockUnits/advisoryLookup/getUnitDetail), cycleCountActions (start/record/submitForReview/approve), purchaseOrderActions (receivePoLine…), transferActions, salesOrderActions, fulfillmentActions, etc. Engine in `packages/inventory/src/lib/` (movements, levels, consume, availability).
- RBAC: resources `inventory`, `vendor`, `purchase_order`, `sales_order`, `stock_transfer`, `stock_location` (create/read/update/delete, MSP-only, Admin-granted). **Location write scoping** (`assertLocationWritable`): tech = own van + home location.
- **No REST API** except SO document/email routes. **No `barcode` column anywhere** (grep: zero hits). No scan UI anywhere in web.
- Web UI: `server/src/app/msp/inventory/*` → components in `packages/inventory/src/components/*`.

### Opportunities / interactions / schedule — map digest
- Opportunities REST **exists**: `server/src/app/api/v1/opportunities/**` (CRUD, win/lose/complete-action, evidence, quotes link, suggestions, EE routes) via `ApiOpportunityController`/`OpportunityService`. **Not in generated OpenAPI spec** (0 paths in `sdk/docs/openapi/*`).
- Missing REST: work queue (`getWorkQueue` action only), timeline (`listOpportunityTimeline` action only), **interactions entirely** (no controller/service/schema — biggest blocker for call logging).
- Interactions: table + `IInteraction` (`type_id`, `contact_name_id`, `client_id`, `ticket_id`, `project_id`, `opportunity_id`, `title`, `notes`, `duration`, `start_time/end_time`, `status_id`, `visibility` default internal). **System type `Call` exists** (icon `phone`, seeded `20241223015715`). Model `packages/clients/src/models/interactions.ts`; actions `packages/clients/src/actions/interactionActions.ts` (`assertMspPermission(user,'interaction',…)`); side-effects in `interactionCreateHelper.ts`.
- Schedule: `schedule_entries` + `schedule_entry_assignees`; `work_item_type` CHECK ('project_task','ticket','interaction','ad_hoc','appointment_request'); REST `/api/v1/schedules*` via `ApiTimeSheetController` — **mobile already consumes it** (`src/api/schedule.ts`). `user_schedule` RBAC: read=own, update=all.
- Queue/discipline: `getWorkQueue` is current-user docket w/ tenant-tz bucketing + why-sentences from `whyComposer.ts` (server-side composer — mobile must never re-derive). `completeNextAction` logs an interaction atomically. Escalation creates ad_hoc calendar blocks (precedent for follow-up entries).
- User-activities feed already surfaces opportunity next actions via `/api/v1/activities` (mobile consumes) — the queue endpoint is still worth it (sections/why-sentences/money framing are queue-specific).
- RBAC: `opportunities` (CRUD, MSP-only), `interaction` (CRUD, MSP-only, seeded `20260622120000` + readd `20260701120000`).

### Workflow-engine state (inventory)
- `packages/inventory/src/lib/inventoryEvents.ts`: 13 event types; published from stockLedger/PO/fulfillment/RMA/SO/dropShip/loaner actions + materials lib.
- Catalog (`20260702150000`): only 4 seeded — STOCK_LOW, PO_RECEIVED, SO_FULFILLED, RMA_CREATED, with `payload.Inventory*.v1` schema refs (schemas in `packages/event-schemas/src/schemas/domain/inventoryEventSchemas.ts`; notification subscriber `server/src/lib/eventBus/subscribers/inventoryNotificationSubscriber.ts`).
- **No `inventory.ts` in `shared/workflow/runtime/actions/businessOperations/`** (opportunities/assets/tickets/clients/scheduling/etc. all have modules). Pattern to mirror: `businessOperations/opportunities.ts` + `registerOpportunityActionsMetadata.test.ts`.
- STOCK_LOW emitted only from fulfillment consume + materials consume — NOT from adjustStock or count approval (parity gap, F066).
- Transfers, counts, RMA status transitions, loaners publish no domain events at all (loaner aging deferred — needs a job).

### API-layer conventions (for the new controllers)
- Pattern: `ApiBaseController` subclasses in `server/src/lib/api/controllers/`, services in `server/src/lib/api/services/`, Zod in `server/src/lib/api/schemas/`, thin route files under `server/src/app/api/v1/**`. Auth middleware `apiAuthMiddleware.ts` (`x-api-key`, optional bearer). Reference implementation: `ApiOpportunityController`; its test coverage is 4 files / 9 tests of schemas/OpenAPI-registry/mocked-service only — NO auth/RBAC/DB assertions, which is why this plan's persistence/RBAC/product-boundary tests are tagged `server-integration`.
- OpenAPI: `sdk/scripts/generate-openapi.ts` → `sdk/docs/openapi/*`; sync skill `alga-openapi-sync` (needs nm-store working dir — already an additional workspace).
